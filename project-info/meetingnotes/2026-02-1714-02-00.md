## **D6 Report Requirements**

Discussed the new requirements for this deliverable beyond what D5 needed:

**Acceptance Criteria:**
- Pick around 5 key user stories and write formal acceptance criteria for them
- Don't need to do all 20+ stories — just demonstrate we know how to write and test against them
- Acceptance criteria must be laid out in the Overleaf for the stories we choose
- Tests must be explicitly linked back to these criteria

### **Testing**

For each testing strategy in the report, follow the structure from the D6 intro lecture — answer three questions:
1. **What** is it? (1-2 sentences)
2. **Why** are you using it? (1-2 sentences)
3. **How** — one piece of evidence (screenshot of test results, not actual code)

No actual code in the report — not copy-pasted, not screenshots of source. Screenshots of test results and outputs are fine.

**Scenario Testing:**
- Must test against all 5 scenarios from our earlier deliverables
- If we haven't finished all functionality for a scenario (likely for calendar/event scenarios), we can defer those scenario tests to D7
- If we defer, we MUST explicitly state that in the report — otherwise it looks like we forgot and we lose marks


**Boundary and Partition Testing:**
- Need to demonstrate both strategies with at least one evidence item each
- Look into what "test coverage" means in our context — code coverage vs input space coverage
- Partition testing: dividing inputs into equivalence classes and testing a representative from each
- Boundary testing: testing at the edges of those partitions

**Regression Testing:**
- Need a dedicated section for regression testing in the report
- Short paragraph describing how we do it — re-running the existing JUnit suite after integrating new modules proves existing functionality isn't broken

### **Definition of Done**

- Need to include a Definition of Done in the report
- A comprehensive checklist of technical and process activities that allows the team to assert a piece of work is truly complete and potentially shippable — consistent across the whole team
